Hackbox speaks a small JSON-over-WebSocket protocol, so **any WebSocket client in any language can host a game** — no SDK required. This guide walks the whole loop: create a room, connect as the host, listen for players, and push UI to their screens.

::callout{icon="i-simple-icons-unity"}
Building in **Unity**? The [hackbox-unity](https://github.com/devanhurst/hackbox-unity) package wraps this protocol in a C#-friendly API. You can still skim this page to understand what it's doing under the hood.
::

Want the bigger picture first? Read [Core concepts](/basics/concepts) for the mental model. Otherwise, dive in.

## 1. Create a room

Your host creates a room with a `POST` to `https://hackbox.ca/api/rooms`. Rooms are ephemeral — they stay open for 24 hours (see [Lifecycle](/advanced/errors-and-lifecycle#room-lifecycle)).

```json
// POST https://hackbox.ca/api/rooms
// Content-Type: application/json

// Body
{
  "hostId": "11111111-2222-3333-4444-555555555555", // A UUID you generate to identify your host. (required)
  "twitchRequired": false // Require players to log in with Twitch before joining. (optional, default: false)
}

// Response
{
  "ok": true,
  "roomCode": "HKBX"
}
```

Hold on to your `hostId` — it's a shared secret. Anyone who connects with it becomes the host of the room.

## 2. Connect as the host

Open a WebSocket to **`wss://hackbox.ca/r/<roomCode>`**, passing your `userId` as a query parameter. Connecting with the room's `hostId` is what makes you the host:

```
wss://hackbox.ca/r/HKBX?userId=11111111-2222-3333-4444-555555555555
```

Every frame on the wire — in both directions — is a JSON envelope:

```json
{ "type": "<event-name>", "payload": <data> }
```

`type` is the event name; `payload` is its data. That's the entire protocol — the rest of these docs just describe each event's payload.

::callout{icon="i-lucide-heart-pulse"}
**Keepalive:** send a `ping` text frame every ~25 seconds; the server replies `pong`. This keeps the connection alive through network idle timeouts. See [Errors & lifecycle](/advanced/errors-and-lifecycle#keepalive).
::

## 3. Listen for your players

As host, you'll receive a `state.host` event whenever the room's roster changes — a player joins, reconnects, or goes offline. The `payload.members` object is keyed by `userId`:

```json
{
  "type": "state.host",
  "payload": {
    "members": {
      "aaaa...": { "id": "aaaa...", "name": "DEV", "online": true },
      "bbbb...": { "id": "bbbb...", "name": "ASH", "online": false }
    }
  }
}
```

Players also send you events when they interact with their screen — `msg` when they submit, and `change` for in-progress edits. Both are covered in [Core concepts](/basics/concepts#events-from-players).

## 4. Send a player some UI

New players see a default "Hang tight!" screen until you push them something. Send a `member.update` to render UI on their device. Here's a single buzzer button:

```json
{
  "type": "member.update",
  "payload": {
    "to": "aaaa...", // a userId, or an array of userIds
    "data": {
      "ui": {
        "main": {
          "components": [
            {
              "type": "Button",
              "props": { "event": "buzz", "label": "Buzz in!", "value": "BUZZ" }
            }
          ]
        }
      }
    }
  }
}
```

::callout{icon="i-lucide-triangle-alert" color="warning"}
**Each `member.update` replaces that player's whole screen.** The `data` is merged onto a blank canvas, not onto what they're currently seeing — so always send the complete UI you want shown. This is the single most important thing to internalize; see [Core concepts](/basics/concepts#state-is-a-full-replacement).
::

You can also force every player's client to reload with `{ "type": "reload" }`.

## 5. Receive your first event

When the player taps **Buzz in!**, your host receives a `msg`:

```json
{
  "type": "msg",
  "payload": {
    "from": "aaaa...",
    "event": "buzz",
    "message": { "event": "buzz", "value": "BUZZ", "ms": 1432 },
    "timestamp": 1769260462112
  }
}
```

`message` is exactly what the component emitted: the `event` name, its `value`, and `ms` — the milliseconds since that component appeared on screen (a free reaction timer). That's the full round trip.

## Migrating from the old socket.io endpoint

Older versions of Hackbox used a socket.io endpoint (`wss://app.hackbox.ca/socket.io`). Hackbox now runs on raw WebSockets, so socket.io clients can no longer connect. The **event names and payloads are unchanged** — only the transport moved:

- Create rooms at `https://hackbox.ca/api/rooms` (was `https://app.hackbox.ca/rooms`).
- Connect to `wss://hackbox.ca/r/<roomCode>?userId=<hostId>` and exchange `{ type, payload }` envelopes instead of socket.io events.
- **Unity hosts:** update to the latest [hackbox-unity](https://github.com/devanhurst/hackbox-unity), which handles the new transport for you.
