Hackbox is a thin real-time relay between **your host** (the game running on a console, laptop, or stream) and **players** (people on their phones). Your host owns all the game logic; Hackbox just routes messages and renders the UI you describe on each player's screen.

## The pieces

| Piece | What it is |
| --- | --- |
| **Host** | Your game. It creates the room and is the single authority connected with the room's `hostId`. |
| **Player** | A person who joins with the room code on their phone. Hackbox's player client renders the UI you send them. |
| **Room** | An ephemeral session identified by a 4-letter code (e.g. `HKBX`). Lives 24 hours. |
| **Relay** | Hackbox itself. It forwards messages between host and players and remembers each player's last screen. |

You never write player-side code. You describe a player's screen as JSON, and Hackbox renders it.

## The envelope

Every frame, in both directions, is a JSON envelope:

```json
{
  "type": "<event-name>",
  "payload": <data>
}
```

There is nothing else to the transport. `type` selects the handler; `payload` carries the data. Send `ping` (a bare text frame, not an envelope) every ~25s to keep the socket alive.

## Member state

When you send a player UI, you're setting their **member state** — a single object with three top-level keys:

```json
{
  "theme": { /* colors and fonts — see Theming */ },
  "ui": {
    "header": { "text": "Round 1" },
    "main": {
      "align": "start",          // "start" | "center" | "end"
      "components": [ /* the UI components, rendered top to bottom */ ]
    }
  },
  "presets": { /* optional reusable component templates — see Presets */ }
}
```

- **`theme`** styles the screen — header and background colors, fonts. See [Theming & layout](/advanced/theming).
- **`ui.header.text`** is the text in the player's header bar.
- **`ui.main.align`** controls vertical alignment of the components (`start`, `center`, or `end`).
- **`ui.main.components`** is the ordered list of [components](/components/overview) to render.
- **`presets`** lets you define component templates once and reuse them. See [Presets](/advanced/presets).

Everything is optional. Anything you omit falls back to a default (dark background, light text, the `Fredoka One` font, an empty header).

### State is a full replacement

This is the most important rule in Hackbox:

::callout{icon="i-lucide-triangle-alert" color="warning"}
Every `member.update` **replaces** the target player's entire state. The `data` you send is merged onto a blank default canvas — **not** onto the player's current screen.
::

In practice: if a player is looking at a question and you send a `member.update` that only contains a new `theme`, their components vanish, because the new state has no `components`. Always send the **complete** screen you want the player to see — header, theme, and components together.

Why it works this way: the relay caches each player's last full state and replays it verbatim when they reconnect, so a player who drops and rejoins lands back on exactly the screen you last sent. A full-replacement model makes that cache simple and correct.

## Events from your host

Send these to the relay:

### `member.update`

Set the state for one or more players.

```json
{
  "type": "member.update",
  "payload": {
    "to": ["aaaa...", "bbbb..."], // a single userId string, or an array of them
    "data": { /* a full member state: theme, ui, presets */ }
  }
}
```

`to` accepts either a single `userId` or an array. Only players who have already joined the room are updated — sending to an unknown id is a no-op.

### `reload`

Force every connected player's client to reload the page. Useful to recover clients after a host restart.

```json
{ "type": "reload" }
```

## Events from players

Your host receives these:

### `state.host`

Sent whenever the roster changes — a player joins, reconnects, or goes offline. `payload.members` is keyed by `userId`:

```json
{
  "type": "state.host",
  "payload": {
    "members": {
      "aaaa...": {
        "id": "aaaa...",
        "name": "DEV",
        "online": true,
        "twitchData": {
          "id": "565...",
          "username": "heydevdev",
          "photo": "https://..."
        }
      },
      "bbbb...": {
        "id": "bbbb...",
        "name": "ASH",
        "online": false
      }
    }
  }
}
```

Each member has an `id`, an uppercased display `name`, and an `online` flag (derived live from the connection — there's no "leave" event; players simply go `online: false`). `twitchData` is present only when the player logged in with Twitch — see [Twitch auth](/advanced/twitch).

### `msg` and `change`

A player interacted with a component. The two events are **identical in shape**; they differ only in intent:

- **`msg`** — the player **submitted**. Act on it.
- **`change`** — work in progress: the player is still editing (e.g. dragging a slider, typing, toggling choices before submit). Useful for live previews or for accepting a player's current selection when a timer expires.

```json
{
  "type": "msg",
  "payload": {
    "from": "aaaa...",
    "event": "answer",
    "message": {
      "event": "answer",
      "value": "B",
      "ms": 1432
    },
    "timestamp": 1769260462112
  }
}
```

- **`from`** — the player's `userId`.
- **`event`** — the component's `event` name, so you can route the message.
- **`message`** — exactly what the component emitted: its `event`, its `value`, and `ms` (milliseconds since the component appeared — a built-in reaction timer).
- **`timestamp`** — server time when the relay received it.

The shape of `value` depends on the component (a string, a number, an array…). Each one is documented in [Components](/components/overview).

## The event catalog at a glance

| Direction | Event | Payload |
| --- | --- | --- |
| host → relay | `member.update` | `{ to, data }` |
| host → relay | `reload` | _(none)_ |
| relay → host | `state.host` | `{ members }` |
| relay → host | `msg` / `change` | `{ from, event, message, timestamp }` |
| relay → player | `state.member` | _(the member state — handled by Hackbox's client)_ |
| relay → player | `reload`, `error` | _(handled by Hackbox's client)_ |

The bottom two rows are between the relay and Hackbox's own player client — you never send or handle them directly, but they're listed so the picture is complete. See [Errors & lifecycle](/advanced/errors-and-lifecycle) for `error` and disconnects.
