Hackbox speaks a small JSON-over-WebSocket protocol. If you're building in **Unity**, the [hackbox-unity](https://github.com/devanhurst/hackbox-unity) package wraps everything below in a Unity-friendly API — start there. Otherwise, any WebSocket client in any language can connect directly by following this guide.

## 1. Create a room

First, your app needs to create a new room. These rooms are ephemeral and will remain open for 24 hours.

To create a room, make a `POST` request to `https://hackbox.ca/api/rooms`.

```json
// Headers
{
  "Content-Type": "application/json"
}

// Body
{
  "hostId": "11111111-2222-3333-4444-555555555555", // A UUID that identifies your host connection. Generate this yourself. (uuid, required)
  "twitchRequired": false // Whether to require players to login with Twitch before joining your room. (boolean, optional, default: false)
}

// Returns
{
  "ok": true,
  "roomCode": "HKBX"
}
```

## 2. Connect via websocket

Now that your room is created, anyone can connect to it via websocket, but _you_ can connect to it as the _host_ by providing the hostId you used when creating the room.

Open a WebSocket to **`wss://hackbox.ca/r/<roomCode>`**, passing your `userId` as a query parameter. Connecting with the room's hostId is what makes you the host:

```
wss://hackbox.ca/r/HKBX?userId=11111111-2222-3333-4444-555555555555
```

Every message on the wire — in both directions — is a JSON envelope of the form:

```json
{ "type": "<event-name>", "payload": <data> }
```

`type` is the event name and `payload` is its data. The sections below describe each event's payload.

> **Keepalive:** send a `ping` text frame every ~25 seconds; the server replies `pong`. This keeps the connection from being dropped by network idle timeouts. (The hackbox-unity package does this for you.)

## 3. Listen for events

Set up handlers for the messages the server sends to your host.

### state.host

`state.host` fires every time the state of the room changes. This includes when a new player joins, and when any player's online status changes. The `payload` carries a `members` object keyed by userId.

```json
{
  "type": "state.host",
  "payload": {
    "members": {
      "11111111-1111-1111-1111-111111111111": {
        "id": "11111111-1111-1111-1111-111111111111",
        "name": "Dev",
        "online": true,
        "twitchData": {
          "id": 56475817,
          "username": "heydevdev",
          "photo": "https://static-cdn.jtvnw.net/jtv_user_pictures/cc0dddae-bbdd-4b1a-a6f7-9a718b4da633-profile_image-300x300.png"
        }
      },
      "22222222-2222-2222-2222-222222222222": {
        "id": "22222222-2222-2222-2222-222222222222",
        "name": "Ash",
        "online": true
      },
      "33333333-3333-3333-3333-333333333333": {
        "id": "33333333-3333-3333-3333-333333333333",
        "name": "Ben",
        "online": false
      }
    }
  }
}
```

### msg

`msg` is fired when a user sends you a message. This is generally done in response to a button press. The `payload` carries the sending user (`from`), the event name, the player's full message, and a timestamp.

```json
{
  "type": "msg",
  "payload": {
    "from": "11111111-1111-1111-1111-111111111111",
    "event": "PickSix",
    "message": {
      "event": "PickSix",
      "value": ["B", "D", "A", "F", "E", "C"]
    },
    "timestamp": 1769260462112
  }
}
```

### change

The `change` event is identical in shape to the `msg` event, however it represents a work in progress. The user hasn't explicitly submitted their answer; they are simply modifying a field. This can be useful if you want to accept a user's current selection when a timer expires.

```json
{
  "type": "change",
  "payload": {
    "from": "11111111-1111-1111-1111-111111111111",
    "event": "PickSix",
    "message": {
      "event": "PickSix",
      "value": ["B", "D", "A"] // Player has only selected three of six answers.
    },
    "timestamp": 1769260462101
  }
}
```

## 4. Send UI payloads

With the listeners set up, you should be receiving the `state.host` payload whenever a new user joins your room on hackbox. But, their screen will only display a welcome message for now. It's up to your host to send payloads to customize their game screens.

Send a `member.update` message. Here's a quick example that will display a single buzzer button on the player's screen:

```json
{
  "type": "member.update",
  "payload": {
    "to": ["11111111-1111-1111-1111-111111111111"], // Array of IDs of all users that should receive this message.
    "data": {
      "ui": {
        "main": {
          "components": [
            {
              "type": "Button",
              "props": {
                "event": "MyFirstEvent",
                "label": "Buzz in",
                "value": "buzz"
              }
            }
          ]
        }
      }
    }
  }
}
```

You can also force every player's client to reload by sending `{ "type": "reload" }`.

If you've sent the `member.update` successfully, your players should see a "Buzz in" button on their screen. If your listeners are set up correctly, when a player taps that button you will receive your first `msg` event whose `payload.message` is:

```json
{
  "event": "MyFirstEvent",
  "value": "buzz"
}
```

## Migrating from the old socket.io endpoint

Earlier versions of hackbox used a socket.io endpoint (`wss://app.hackbox.ca/socket.io`). hackbox now runs on raw WebSockets, so socket.io clients can no longer connect. The **event names and payloads are unchanged** — only the transport moved:

- Create rooms at `https://hackbox.ca/api/rooms` (was `https://app.hackbox.ca/rooms`).
- Connect to `wss://hackbox.ca/r/<roomCode>?userId=<hostId>` and exchange `{ type, payload }` envelopes instead of socket.io events.
- **Unity hosts:** update to the latest [hackbox-unity](https://github.com/devanhurst/hackbox-unity), which handles the new transport for you.
