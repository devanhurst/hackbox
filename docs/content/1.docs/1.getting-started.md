## 1. Create a room

First, your app needs to create a new room. These rooms are ephemeral and will remain open for 24 hours.

To create a room, make a `POST` request to `https://hackbox.ca/api/rooms`.

```json
// Headers
{
  "Content-Type": "application/json"
}

// Body
{
  "hostId": "11111111-2222-3333-4444-555555555555", // A UUID that identifies your host connection. Generate this yourself. (uuid, required)
  "twitchRequired": false // Whether to require players to login with Twitch before joining your room. (boolean, optional, default: false)
}

// Returns
{
  "ok": true,
  "roomCode": "HKBX"
}
```

## 2. Connect via websocket

Now that your room is created, anyone can connect to it via websocket, but _you_ can connect to it as the _host_ by providing the hostId you used when creating the room.

The recommended way to connect is the official **`@hackbox/client`** SDK, which manages the connection (reconnects, keepalive) and gives you a familiar event API:

```bash
npm install @hackbox/client
```

```ts
import { createHackboxSocket } from "@hackbox/client";

const socket = createHackboxSocket({
  host: "hackbox.ca",
  roomCode: "HKBX", // the roomCode returned from POST /api/rooms
  userId: "11111111-2222-3333-4444-555555555555", // the same hostId you created the room with — this is what makes you the host
});
```

> **Not using JavaScript?** You can connect any WebSocket client directly — see
> [Connecting without the SDK](#connecting-without-the-sdk) below.

## 3. Listen for events

Now it's time to set up listeners to handle the messages sent to your host.

### state.host

`state.host` fires every time the state of the room changes. This includes when a new player joins, and when any player's online status changes. The payload is an object of `members` keyed by userId.

```ts
socket.on("state.host", (payload) => {
  // payload.members is keyed by userId
});
```

```json
{
  "members": {
    "11111111-1111-1111-1111-111111111111": {
      "id": "11111111-1111-1111-1111-111111111111",
      "name": "Dev",
      "online": true,
      "twitchData": {
        "id": 56475817,
        "username": "heydevdev",
        "photo": "https://static-cdn.jtvnw.net/jtv_user_pictures/cc0dddae-bbdd-4b1a-a6f7-9a718b4da633-profile_image-300x300.png"
      }
    },
    "22222222-2222-2222-2222-222222222222": {
      "id": "22222222-2222-2222-2222-222222222222",
      "name": "Ash",
      "online": true
    },
    "33333333-3333-3333-3333-333333333333": {
      "id": "33333333-3333-3333-3333-333333333333",
      "name": "Ben",
      "online": false
    }
  }
}
```

### msg

`msg` is fired when a user sends you a message. This is generally done in response to a button press. The payload carries the sending user (`from`), the event name, the player's full message, and a timestamp.

```ts
socket.on("msg", (payload) => {
  // payload: { from, event, message, timestamp }
});
```

```json
{
  "from": "11111111-1111-1111-1111-111111111111",
  "event": "PickSix",
  "message": {
    "event": "PickSix",
    "value": ["B", "D", "A", "F", "E", "C"]
  },
  "timestamp": 1769260462112
}
```

### change

The `change` event is identical in shape to the `msg` event, however it represents a work in progress. The user hasn't explicitly submitted their answer; they are simply modifying a field. This can be useful if you want to accept a user's current selection when a timer expires.

```json
{
  "from": "11111111-1111-1111-1111-111111111111",
  "event": "PickSix",
  "message": {
    "event": "PickSix",
    "value": ["B", "D", "A"] // Player has only selected three of six answers.
  },
  "timestamp": 1769260462101
}
```

## 4. Send UI payloads

With the listeners set up, you should be receiving the `state.host` payload whenever a new user joins your room on hackbox. But, their screen will only display a welcome message for now. It's up to your host to send payloads to customize their game screens.

Here's a quick example payload that will display a single buzzer button on the player's screen. Send it with the `member.update` event:

```ts
socket.emit("member.update", {
  to: ["11111111-1111-1111-1111-111111111111"], // Array of IDs of all users that should receive this message.
  data: {
    ui: {
      main: {
        components: [
          {
            type: "Button",
            props: {
              event: "MyFirstEvent",
              label: "Buzz in",
              value: "buzz",
            },
          },
        ],
      },
    },
  },
});
```

You can also send `reload` to force every player's client to reload:

```ts
socket.emit("reload");
```

If you've sent the `member.update` successfully, your players should see a "Buzz in" button on their screen. If your listeners are set up correctly, when a player taps that button you will receive your first `msg` event whose `message` is:

```json
{
  "event": "MyFirstEvent",
  "value": "buzz"
}
```

## Connecting without the SDK

The SDK is just a thin wrapper over a standard WebSocket — you can connect any client, in any language, directly.

Connect to **`wss://hackbox.ca/r/<roomCode>`**, passing your `userId` as a query parameter (use your hostId to connect as the host):

```
wss://hackbox.ca/r/HKBX?userId=11111111-2222-3333-4444-555555555555
```

Every message on the wire is a JSON envelope of the form `{ "type": <event>, "payload": <data> }`. The `type` is the event name, and the `payload` is exactly what the SDK hands your listener. For example, a `msg` arrives as:

```json
{
  "type": "msg",
  "payload": {
    "from": "11111111-1111-1111-1111-111111111111",
    "event": "PickSix",
    "message": { "event": "PickSix", "value": ["B", "D", "A", "F", "E", "C"] },
    "timestamp": 1769260462112
  }
}
```

…and you send a `member.update` the same way:

```json
{
  "type": "member.update",
  "payload": {
    "to": ["11111111-1111-1111-1111-111111111111"],
    "data": { "ui": { "main": { "components": [/* … */] } } }
  }
}
```

To keep the connection alive through NAT/idle timeouts, send a `ping` text frame every ~25 seconds; the server replies `pong`. (The SDK does this for you.)

## Migrating from the old socket.io endpoint

Earlier versions of hackbox used a socket.io endpoint
(`wss://app.hackbox.ca/socket.io`). hackbox now runs on raw WebSockets, so
socket.io clients can no longer connect. The **event names and payloads are
unchanged** — only the transport moved:

- Create rooms at `https://hackbox.ca/api/rooms` (was `https://app.hackbox.ca/rooms`).
- Replace your socket.io client with `@hackbox/client` (or a raw WebSocket per the
  section above). Your `on(...)` / `emit(...)` event handling stays the same.
