This page covers the operational edges of a Hackbox connection: keeping it alive, what happens when it drops, and how long a room lasts.

## Keepalive

Browsers and mobile networks silently drop idle WebSocket connections — often after 60–90 seconds. To prevent that, **send a `ping` text frame every ~25 seconds**:

```
→ ping
← pong
```

`ping` is a bare text frame, not a `{ type, payload }` envelope. The server answers `pong` at the edge (it doesn't even wake the room), and the round trip keeps the network path warm. If you skip keepalives, expect occasional unexplained disconnects on flaky networks.

::callout{icon="i-simple-icons-unity"}
The [hackbox-unity](https://github.com/devanhurst/hackbox-unity) package sends keepalives for you.
::

## Error frames

When the relay refuses or terminates a connection, it sends an `error` envelope and then closes the socket with a **close code of `4000`**:

```json
{
  "type": "error",
  "payload": { "message": "This room is closed." }
}
```

A close code **≥ 4000 is terminal** — the relay doesn't want this connection, so **do not reconnect**. Surface the `message` to the user instead. The reasons:

| Message | Why |
| --- | --- |
| `This room does not exist.` | No room with that code (never created, or already expired/deleted). |
| `Missing user id.` | The connection didn't include a `userId` query param. |
| `This room is closed.` | The room is closed to new members and this user isn't already in it. |
| `Please log in with Twitch before joining this room.` | The room requires [Twitch auth](/advanced/twitch) and the player hasn't authenticated. |
| `You have connected from another device.` | Another connection opened with the same `userId`; the older one is evicted (one live connection per user). |
| `This room has expired.` | The room hit its 24-hour limit while you were connected. |
| `This room has been deleted.` | The room was removed by an admin. |

A close with a code **below 4000** (e.g. `1006`) is a transient network drop, not a rejection — reconnect normally.

::callout{icon="i-lucide-info"}
Most of these target **players**, and Hackbox's player client handles them for you. They matter when you're writing your own client — including the host connection, which can receive `You have connected from another device.`, `This room has expired.`, and `This room has been deleted.`
::

## Reconnecting and state replay

For any non-terminal drop (close code < 4000), reconnect by opening the same `wss://hackbox.ca/r/<roomCode>?userId=<id>` URL. A small exponential backoff (a few hundred milliseconds, growing to ~1s) is plenty.

The relay keeps each player's **last screen** and replays it the moment they reconnect:

- A **player** who reconnects immediately receives a `state.member` with the exact UI you last sent them — they land back where they were, not on a blank screen. You don't need to re-push anything.
- A **host** who reconnects receives a fresh [`state.host`](/basics/concepts#events-from-players) roster. Your game logic lives in your host, so rebuild any in-memory game state from your own source of truth.

Because the relay only remembers the *last* state per player, this is another reason every [`member.update` is a full replacement](/basics/concepts#state-is-a-full-replacement) — the cached screen is always complete and ready to replay.

## Room lifecycle

Rooms are ephemeral:

- A room is created by `POST /api/rooms` and exists the moment that call returns a `roomCode`.
- It lives for **24 hours**, then self-destructs: every connection is closed with `This room has expired.` and the code is freed for reuse.
- There's no "delete my room" call in the public API — just let it expire, or create a fresh one for your next session.

Plan for the 24-hour window: it's generous for a game night, but a room code is not a permanent address. Create a room when a session starts, and don't assume yesterday's code still works.
